# 設計フル実行スキル

`.rulesync/rules/design-workflow.md` に定義された全フェーズを、順次・**半自律的に**実行する。
**ドメインロジックに関わる判断は必ずユーザーに確認を取り、推測では進めない。**

## 前提として読み込む context

オーケストレータ。**直接の必読 context は持たない** (各サブスキルが必要に応じて読む)。

各フェーズで起動するサブスキル (`/spec-create`, `/check-convention`, `/spec-resolve`, `/spec-diagrams`, `/spec-tests`, `/spec-track`, `/doc-review`) が、それぞれの「前提として読み込む context」セクションに従って **必要な context だけを** 選択的に読む。

**読み込まない**: spec-full 自体が context を一括ロードしてはならない。token 浪費の温床になる。

詳細は `docs/context/README.md` の対応表を参照。

## 入力

- `$ARGUMENTS`: 要求番号 / 既存設計書のパス
- 引数なし: ユーザーに対象を確認する

## 中核原則

### 共通ガード (全フェーズ共通の原則)

- **未確定ドメイン論点が 1 つでも残っている場合、次フェーズへ進まず停止してユーザーに質問する**
- **各フェーズ完了後、必ず次フェーズ提案 → ユーザー同意 → 実行 の順で進める** (自動遷移しない)
- **`/doc-review` はフェーズ 2 以降の完了時に必須。スキップ禁止** (フェーズ 1 は設計書が未生成のため対象外)
- **確定済みの判断 (解決済みの論点) を上書きしない**

### ハンドオフ基準 (必ず人間判断を仰ぐ)

以下検知時は**停止してユーザーに質問する**。推測禁止。

1. **ドメインロジック**: 業務意味が複数解釈できる / 要件未明記の振る舞い
2. **データ整合性**: 既存データとの整合リスク、破壊的マイグレーションの可能性
3. **認可**: ロール別アクセス可否が要件から読めない
   - 加えて、**マスター認可マトリクス (`docs/context/authorization/matrix.md`) と要求 / 設計書下書きの認可記述に 1 セルでも差分がある場合は即停止**
   - 詳細は `.rulesync/rules/authorization-matrix.md` を参照
4. **トランザクション境界**: 複数 Op 横断の失敗補償が不明確
5. **既存規約との不整合**: `/check-convention` 結果と要件が衝突
6. **OIDC 標準逸脱**: id-core が OP として OIDC 標準から逸脱する可能性
7. **レビュー CRITICAL / HIGH**: 自動修正禁止。対応方針を確認
8. **論点テーブルに未決定が残存している状態で次フェーズ進行の判断**

停止テンプレ:

```
【判断が必要です: {フェーズ名}】
論点: {何が不明か}
選択肢:
  A) {案 A} — メリット / デメリット
  B) {案 B} — メリット / デメリット
  (推奨: A / なし)
このまま進めますか？選択肢を教えてください。
```

## 実行手順

### 1. 事前確認

- 対象要求 / 設計書のパスを確定
- 既存設計書がある場合、**「設計フェーズ状況」テーブル**を参照して現在フェーズを検出
  - `進行中` → そのフェーズから再開
  - `完了` → **レビュー未了扱い**。同フェーズの `/doc-review` から再開 (ただしフェーズ 1 はレビュー対象外のため `完了` を終端とし、次フェーズに進める)
  - `保留` → 備考の解消をユーザーに確認。解消後は `進行中` に戻し、同フェーズを再実行
  - 複数 `保留` がある場合は、フェーズ番号の小さい順に解消する
  - 上記いずれもない場合、直近の `未着手` から開始

### 2. フェーズ実行ループ

各フェーズのサイクル:

```
[ユーザー同意] → [スキル実行] → [ハンドオフ判定] → [doc-review] → [指摘対応判定]
  → [PhaseStatus 更新] → [次フェーズ提案] → [ユーザー同意] → …
```

| フェーズ | スキル | 必須停止タイミング |
|---|---|---|
| 1. 起票 | (要求確認) | 要件曖昧なら停止 |
| 2. 下書き | `/spec-create` | 要求との矛盾で停止 |
| 3. 規約確認 | `/check-convention` | 要件 - 規約衝突で停止 |
| 4. 論点解決 | `/spec-resolve` (全件) | **全論点で毎回停止** (必須) |
| 5. DB 設計 | (手動記載) | スキーマ破壊的変更・型選定で停止 |
| 6. API 設計 | (手動記載) | エンドポイント粒度・認可不明で停止 |
| 7. 認可設計 | (手動記載 + マトリクス突合) | ロール要件不明で停止 |
| 8. 図の生成 | `/spec-diagrams` | 未確定ドメイン論点があれば停止 |
| 9. テスト設計 | `/spec-tests` | 観点漏れ疑いで停止 |
| 10. 差分整理 | `/spec-track` | 未確定論点があれば停止 |

### 3. レビュー (各フェーズ完了時)

- フェーズ 1 以外、完了時に `/doc-review` を必ず実行
- 結果を重大度別に要約
- **CRITICAL / HIGH: 自動修正禁止**。方針をユーザーに確認
- **自動修正を許可する対象は以下に限定** (これ以外は重大度に関係なくユーザー確認):
  - typo (誤字脱字)
  - リンク切れ
  - 見出し整形 (Markdown 見出しレベル・番号揃え)
  - 表組の体裁 (カラム整形・空白調整)
- 対応後、再度 `/doc-review` を回して CRITICAL / HIGH ゼロを確認

### 4. PhaseStatus 更新

フェーズ完了 (レビュー指摘対応含む) ごとに「設計フェーズ状況」テーブルを更新:

- 作業開始時: `進行中` / 最終更新日記入
- スキル完了: `完了`
- レビュー指摘対応完了: `レビュー済`
- ユーザー判断待ち: `保留` + 備考に理由

### 5. 進捗報告

各フェーズ完了時:

```
✅ フェーズ N: {フェーズ名} {状態}
  - 変更箇所: {概要}
  - レビュー: CRITICAL 0 / HIGH 0 / MEDIUM {n} / LOW {n}
  - PhaseStatus: {状態}
次フェーズ: {N+1. 名前} に進みますか？(Y / 後で / 中断)
```

## 終了条件

- フェーズ 10 が `レビュー済`
- 最終レビューで CRITICAL / HIGH ゼロ
- 論点テーブル未決定ゼロ
- ユーザーにサマリー + 実装フェーズ起動の提案を提示

## 禁止事項

- ドメインロジックを推測で決める
- レビュー CRITICAL / HIGH を独断で修正
- 論点テーブルに未決定を残したままフェーズ 5 以降に進む
- `/doc-review` のスキップ
- ユーザー確認なしの次フェーズ移行
- 確定済みの判断 (解決済みの論点) の上書き
- ユーザー応答待ちタイムアウト (「反応がないので進めます」)
- typo・リンク切れ・見出し整形・表組の体裁 以外の自動修正
