# 認可マトリクス運用ルール

## 正本 (Source of Truth)

このリポジトリの `docs/context/authorization/matrix.md` を**認可マトリクスの唯一の正本**とする。

- 設計書 (`docs/specs/*/index.md`) の認可表は**正本の参照コピー**であり、独自判断で拡大/縮小してはならない
- 認可ポリシー実装 (id-core の認可エンドポイント / マイグレーション / フロントエンド側のロール別 UI 制御) も正本の権限を超えてはならない

## 絶対不変扱い (推測による変更禁止)

**マスターのセル (ロール × 機能) を推測で書き換えることは禁止。**

設計書作成 / 論点解決 / 認可ポリシー生成のいずれのスキルでも、以下を厳守:

1. マスターで `空欄` (拒否) のセルを `o` / `▲` に拡大する提案は**ユーザー判断必須**
2. マスターで `o` のセルを縮小する提案も**ユーザー判断必須**
3. マスターに未記載の機能・ロールを追加する場合も**ユーザー判断必須**
4. 要件記述とマスターで権限記述が食い違う場合は、**どちらが正か決めるのはユーザー** (スキルは停止して提示)

## 必須のハンドオフ基準

以下検知時は **即停止してユーザーに質問する**。推測禁止。

- 要求 / 設計書下書きの認可記述が、マスター `docs/context/authorization/matrix.md` と**1セルでも異なる**
- マスターに未定義の新ロール・新機能を扱う必要がある
- `▲` (条件付き許可) の条件 (所有権 / 組織 / 経路等) がマスターから読み取れない
- 複数ロールが同一機能に対して異なる条件付き許可を持つ場合の優先順位が不明

停止テンプレ:

```
【認可マトリクス差分を検出しました】
対象: {機能名}
ロール: {ロール名}
マスター: {空欄/o/▲}
提示元: {要求/設計下書き} = {空欄/o/▲}
選択肢:
  A) マスターに合わせる (現状維持)
  B) マスターを {値} に変更する (要: セキュリティ判断)
     → 承認された場合は /spec-resolve の「マスター変更の正規フロー」で反映
       (ADR 起票 → 正本 docs/context/authorization/matrix.md 更新 → 認可ポリシー再生成)
  C) 保留 (論点テーブルに追加)
どれにしますか？
```

## スキル別の適用

| スキル | マスター突合義務 | 違反時の挙動 |
|---|---|---|
| `spec-create` | 要件・対象機能のいずれかに認可記述があればマスターと突合 | 差分あり → 停止 |
| `spec-resolve` | 認可に関わる論点決定前にマスター参照 | 差分あり → 停止 |
| `spec-full` | 上記を全フェーズで強制 | 差分あり → 停止 |
| `requirements-create` | ドラフト時に認可記述があれば本ルールに従う | 差分あり → 停止 |
| `requirements-resolve` | 認可に関わる論点決定前にマスター参照 | 差分あり → 停止 |
| `requirements-full` | 上記を全フェーズで強制 | 差分あり → 停止 |
| `check-convention` | 既存マトリクスの形式規約を報告 | 差分そのものは検出対象外だが報告 |

## マスター変更の正規フロー

マスターの変更が必要と判断された場合 (ユーザー承認後のみ):

1. 設計書内に「**認可マスター変更提案**」セクションを新設し、差分 (before/after) ・理由・影響範囲を記載
2. `/adr-create` で ADR を起票 (セキュリティ影響のため記録必須)
3. ユーザーが `docs/context/authorization/matrix.md` を直接更新 (または明示承認後にスキルが更新)
4. 更新後の正本をもとに認可ポリシー実装を再生成

設計書側のマトリクス / 認可ポリシー実装を先に書き換えてはならない。
