# バグ報告 → Issue 起票スキル

バグの詳細を整理し、Issue テンプレートに沿った本文を生成して起票する。

## ワークフロー

```
バグ情報の収集 → 構造化 → ドラフト提示 → ユーザー承認 → Issue 起票
```

## 手順

### 1. バグ情報の収集

`$ARGUMENTS` またはユーザーの発言からバグの詳細を把握する。
不足している情報があれば質問する:

- 何が起きたか (期待と実際の差分)
- 再現手順
- スクリーンショットやログ
- 影響範囲と頻度
- 環境 (ブラウザ、OS、ビルドハッシュ等)
- 影響対象 (`core/` か `examples/...` か)

### 2. 情報の補強

会話の中でコードやログが共有されていれば、関連情報を整理する:

- エラーメッセージの抽出
- 再現手順の具体化
- 影響を受ける画面・機能の特定

### 3. ドラフト作成・提示 (必須 — スキップ禁止)

`.github/ISSUE_TEMPLATE/bug.md` (あれば) のセクションに沿って本文を作成し、
**必ずユーザーに全文を提示する**。ラベルの提案も併記する:

- 種別: `種別:バグ` (固定)
- 状態: `状態:要求分析中` (固定、原因分析・対応方針検討の開始)
- 優先度: 影響度から提案 (`優先:高` / `優先:中` / `優先:低`)
- 対象: 影響範囲から提案 (`対象:基盤` / `対象:サーバー` / `対象:画面`)

> **ガード**: ドラフトをユーザーに提示せずに起票してはならない。
> ユーザーが「そのまま起票して」と言った場合でも、ドラフトを表示した上で起票する。

### 4. マイルストーン特定 (必須)

詳細は `.rulesync/rules/issue-traceability.md` 参照。

1. 既存マイルストーン一覧を取得:

   ```bash
   gh api repos/{owner}/{repo}/milestones --jq '.[] | "\(.number) \(.title)"'
   ```

2. バグの影響範囲から該当マイルストーンを推定し、ユーザーに確認:
   - 既に実装済みの機能のバグ → その機能のマイルストーン
   - 該当マイルストーンが特定できない場合はユーザーに選択を委ねる

### 5. Issue 起票

ユーザーの承認後、`gh issue create` で起票する (**マイルストーン紐付け必須**)。

```bash
gh issue create \
  --title "{タイトル}" \
  --body "{本文}" \
  --milestone "M{N}.{X}: {マイルストーンタイトル}" \
  --label "種別:バグ" \
  --label "状態:要求分析中"
```

起票後、Issue 番号と URL をユーザーに報告する。

## 注意

- 再現手順は番号付きリストで具体的に書く
- スクリーンショットがあれば Markdown の画像記法で埋め込む
- ログは長い場合 `<details>` で折りたたむ
- `状態:要求分析中` は起票=原因・対応方針分析の開始宣言。
  `requirements-publish` 成功時に `状態:設計着手OK` に切り替わる
