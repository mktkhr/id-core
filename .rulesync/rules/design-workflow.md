# 設計ワークフロー

設計書を作成する際の標準的な進行順序。各フェーズが完了したら、次フェーズのスキル実行を提案する。

## フェーズ一覧

全フェーズ共通: **各フェーズ完了後に `/doc-review` でセルフレビューし、指摘対応後に次フェーズへ進む。**

| #   | フェーズ   | 作業                                                                                                             | 完了条件                                | 次フェーズの推奨スキル              |
| --- | ---------- | ---------------------------------------------------------------------------------------------------------------- | --------------------------------------- | ----------------------------------- |
| 1   | 起票       | 要件 / 関連資料の読み取り                                                                                        | 要件が把握できた                        | `/spec-create` で雛形生成           |
| 2   | 下書き     | テンプレートから設計書の骨格作成                                                                                 | 関連資料・要件の解釈が記載された        | `/check-convention` で既存規約確認  |
| 3   | 規約確認   | 既存 DB/API/エラー規約の調査                                                                                     | 規約が設計書に反映された                | `/spec-resolve` で論点を1つずつ解決 |
| 4   | 論点解決   | 未決定事項を 1 つずつ議論・決定                                                                                  | 未確定事項ゼロ                          | 次フェーズへ                        |
| 5   | DB 設計    | ER 図・DDL・シードデータ作成                                                                                     | DDL が書ける状態                        | 次フェーズへ                        |
| 6   | API 設計   | エンドポイント・JSON 構造・エラーコード                                                                          | 全エンドポイントが定義された            | (手動記載) → 認可設計 (フェーズ 7)  |
| 7   | 認可設計   | `docs/context/authorization/matrix.md` から本スコープ分を参照コピー、ロール別 UI 制御 / 認可ポリシーに落とし込む | 認可マトリクスが完成                    | `/spec-diagrams` で図の生成         |
| 8   | 図の生成   | フローチャート + シーケンス図                                                                                    | 全ドメイン操作の図がある                | `/spec-tests` でテスト観点          |
| 9   | テスト設計 | data-testid + テスト観点                                                                                         | バックエンド / E2E のテスト観点が揃った | `/spec-track` で差分追記            |
| 10  | 差分整理   | 既存資料からの変更点を最新化                                                                                     | 差分セクションが最新                    | 設計完了                            |

## 各フェーズの進行フロー

```
フェーズN作業完了 → /doc-review (セルフレビュー) → 指摘対応 → 次フェーズ提案
```

- フェーズの作業が完了したら、ユーザーに確認せず**自動的に** `/doc-review` を実行する
- レビュー指摘を対応した後、「指摘対応完了です。次は{次フェーズ}に進みますか？」と提案する
- フェーズ 1 (起票) のみ設計書がまだ存在しないためレビュー不要

## 提案のタイミング

以下の条件を検知したら、まずセルフレビューを実行し、指摘対応後に次フェーズのスキルを提案する:

- **論点が 1 つ解決した** → 「残り{N}件です。次の論点に進みますか？」 + 全解決なら自動レビュー
- **セルフレビューの指摘を全て対応した** → 「指摘対応完了です。次は{次フェーズ}に進みますか？」
- **DDL セクションが記載された** → 自動レビュー
- **API セクションが記載された** → 自動レビュー → 指摘対応後に認可設計を提案
- **認可マトリクスが完成した** → 自動レビュー → 指摘対応後に `/spec-diagrams` を提案
- **テスト観点が記載された** → 自動レビュー → 指摘対応後に `/spec-track` を提案

## 注意事項

- フェーズは厳密な順序ではなく、ユーザーの判断で前後してよい
- 提案は「〜しますか？」の形で行い、強制しない
