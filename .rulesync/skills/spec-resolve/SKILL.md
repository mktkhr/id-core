# 論点解決スキル

設計書 (`docs/specs/` 配下) の「設計時の論点」テーブルから論点を 1 つ取り上げ、詳細を説明し、ユーザーの決定を設計書に反映する。

## 前提として読み込む context

**必須** (必ず読む):

- 対象の `docs/specs/{N}/index.md` — 論点解決対象 (該当論点セクションのみで足りる場合は部分読みでも可)

**条件付き** (論点の領域に応じて該当時のみ読む):

- 論点が**認可関連**: `docs/context/authorization/matrix.md` (突合必須・推測禁止)
- 論点が**DB / API**: `docs/context/backend/{registry,conventions}.md`
- 論点が**画面 / フロント**: `docs/context/frontend/{registry,conventions}.md`
- 論点が**テスト**: `docs/context/testing/{backend,e2e}.md`
- 論点が**技術スタック / 全体構成**: `docs/context/app/architecture.md`

**読み込まない**: 論点に**関係ない領域**の context は絶対に読まない。

詳細は `docs/context/README.md` の対応表を参照。

## 入力

- `$ARGUMENTS` から対象の論点番号や内容を特定する
- 引数なし: 未決定の論点一覧を表示してユーザーに選択を促す

## 実行手順

### 0. 認可関連論点の前提チェック (該当時必須)

対象の論点が認可 (ロール別アクセス・所有権・権限表) に関わる場合、**先にマスター正本と突合する**。
詳細は `.rulesync/rules/authorization-matrix.md`。

1. `docs/context/authorization/matrix.md` (認可マスター正本) の該当セルを確認
2. 論点の選択肢がマスターの現状セル値と一致するか検証
3. マスターを**変更する方向の決定** (空欄 → o/▲、o → 縮小、新規ロール等) を提示する場合は、**必ず「マスター変更が必要」である旨を明示し、ユーザーの承認を求める**
4. 承認前にマスターを書き換えてはならない。書き換え指示を設計書側にも残さない (「マスター修正が必要」の注記のみ許容)
5. **承認後は `.rulesync/rules/authorization-matrix.md` の「マスター変更の正規フロー」を必ず実行**:
   - 設計書内に「認可マスター変更提案」セクションを新設 (差分 before/after・理由・影響範囲)
   - `/adr-create` で ADR 起票 (セキュリティ影響の記録)
   - ユーザーが `docs/context/authorization/matrix.md` を直接更新 (または明示承認後)
   - 更新後の正本に従って認可ポリシー実装側を再生成

### 1. 論点の詳細説明

対象の論点について以下を提示する:

- 何が決まっていないか
- 選択肢とそれぞれのメリット・デメリット
- 他の論点との依存関係
- 推奨案 (ある場合)

### 2. ユーザーの決定を待つ

ユーザーの回答を待つ。追加の質問があれば回答する。

### 3. 設計書に反映

決定した内容を以下の全箇所に一括反映する:

1. **設計時の論点テーブル**: 「決定」列に結果を記入
2. **ER 図**: 関連するカラム・制約を更新
3. **DDL**: 関連する SQL 定義を更新
4. **ドメイン操作**: 関連する Op 定義を更新
5. **API 仕様**: 関連するエンドポイント・リクエスト・レスポンスを更新
6. **未確定事項**: 解決済みなら「解決済みの論点」に移動

### 4. 変更履歴に追記

変更した内容を「変更履歴」セクションに追記する (`/spec-track` と同等の処理)。

### 5. 残り論点の報告

未決定の論点数と一覧を報告する。
