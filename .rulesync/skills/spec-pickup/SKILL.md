# 設計ピックアップスキル

**要求 Issue を「設計担当として引き取る」ための専用入口。**
要求の完成度を検証し、設計着手可能と判断されたときだけ `/spec-create` を起動する。

## 背景

要求の浅い Issue が設計フェーズに流れるのを構造的に防ぐ。
このスキルは「設計担当の受け取り判断」を自動化する。
(個人開発でも、要求と設計のフェーズを意識的に切ることで品質を確保する)

## 入力

- `$ARGUMENTS` から Issue 番号を特定する
- 引数なし: ユーザーに Issue 番号を確認する

## 手順

### 1. Issue 取得とラベル確認

1. `gh issue view {Issue 番号} --json number,title,body,labels,state`
2. ラベルを確認:
   - `状態:設計着手OK` ラベルなし → **差し戻し (ケース A)**

### 2. 要求文書の存在確認

1. `docs/requirements/{Issue 番号}/index.md` が存在するか確認
2. 存在しない → **差し戻し (ケース B)**

### 3. 受け取り側再検証 (validate)

1. `/requirements-validate` を内部実行
2. 不合格 → **差し戻し (ケース C)**

### 4. 受け取り側再検証 (review) + ゲート判定

1. `/requirements-review` を内部実行し CRITICAL / HIGH / MEDIUM 件数を取得
2. ゲート判定:

| 件数                                   | 判定                                     |
| -------------------------------------- | ---------------------------------------- |
| CRITICAL ≥ 1 または HIGH ≥ 1           | **差し戻し (ケース D)**                  |
| MEDIUM ≥ 3                             | **差し戻し (ケース E)**                  |
| CRITICAL=0 かつ HIGH=0 かつ MEDIUM ≤ 2 | **受け取り可** (MEDIUM は設計側で微調整) |

### 5. 受け取り可の場合

1. ユーザーに引き取り確認:
   - 「残課題 MEDIUM {件数} 件 を設計側で調整しつつ進みますか? (Y/N)」
   - N なら差し戻しを提案
2. Y なら以下を実行:
   - `/spec-create {Issue 番号}` を起動 → `docs/specs/{Issue 番号}/index.md` 初版生成
   - 残 MEDIUM 項目を設計書の「設計時の論点」テーブルに転記
3. ピックアップ記録を Issue コメントとして自動投稿 (後述)

### 6. 差し戻しの場合 (共通処理)

**自動で Issue コメントは投稿しない。** ユーザーに以下を提示:

```
【設計ピックアップを中断しました】

Issue: {Issue URL}
差し戻し理由: {ケース A/B/C/D/E の説明}
不足項目:
  - {具体的な項目 1}
  - {具体的な項目 2}

要求側で修正を行ってください。
```

ラベル操作:

```bash
gh issue edit {Issue 番号} \
  --add-label "状態:要求差し戻し" \
  --remove-label "状態:設計着手OK"
```

## 差し戻しケース定義

| ケース | 条件                                               | 説明                           |
| ------ | -------------------------------------------------- | ------------------------------ |
| A      | `状態:設計着手OK` ラベルなし                       | `/requirements-publish` 未実行 |
| B      | `docs/requirements/{N}/index.md` 不在              | 要求文書未作成                 |
| C      | `requirements-validate` 不合格                     | 設計着手条件を満たさない       |
| D      | `requirements-review` CRITICAL ≥ 1 または HIGH ≥ 1 | 重大な不足/矛盾                |
| E      | `requirements-review` MEDIUM ≥ 3                   | 中程度の調整項目過多           |

## ピックアップ記録 (Issue コメント自動投稿)

受け取り可判定時のみ、以下のコメントを Issue に自動投稿する:

```bash
gh issue comment {Issue 番号} --body "$(cat <<'EOF'
## 🤖 設計ピックアップ記録 (AI 自動生成)

このコメントは `/spec-pickup` スキルにより AI が自動生成・投稿しています。

- ピックアップ日時: {YYYY-MM-DD HH:MM}
- 設計担当: {git config user.name}
- 設計書: `docs/specs/{Issue 番号}/index.md` を生成
- ゲート結果:
  - validate: 合格
  - CRITICAL: 0 / HIGH: 0 / MEDIUM: {件数}
- 残課題 (設計側で調整):
  - {MEDIUM 項目のサマリ}

設計フェーズに移行します。以降の進捗は設計書と関連 PR を参照してください。
EOF
)"
```

## 次工程の案内

受け取り完了後、`/spec-full` または `/spec-resolve` に進むよう案内する。

## 注意

- **ゲートは厳格に適用する。ユーザーが「そのまま進めて」と言っても閾値を緩めない。**
- 差し戻し時は Issue コメントを自動投稿しない (要求側への連絡はユーザー自身が行う)
- 受け取り時のコメント投稿は自動で行い、必ず「AI 自動生成」である旨を明示する
- ラベル運用:
  - 受け取り可 → `状態:設計着手OK` は残したまま (設計フェーズ中の識別用)
  - 差し戻し → `状態:要求差し戻し` を付与、`状態:設計着手OK` を外す
