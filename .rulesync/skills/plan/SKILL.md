# 設計計画スキル

コードを書く前に、設計タスクの包括的な計画を作成する。
設計ワークフロー (`.rulesync/rules/design-workflow.md`) と連携し、フェーズの進行を計画する。

## 前提として読み込む context

**必須** (必ず読む):
- `docs/context/app/architecture.md` — 計画立案の前提となる全体像

**条件付き** (該当する場合のみ読む):
- 既存の関連要求あり: `docs/requirements/{N}/index.md`
- 既存の関連設計書あり: `docs/specs/{N}/index.md`

**読み込まない**: backend/frontend/testing 配下の詳細 context は計画段階では不要。具体化は後続フェーズの skill が行う。

詳細は `docs/context/README.md` の対応表を参照。

## このスキルの機能

1. **要求の再確認** — 何を設計する必要があるかを明確化
2. **現状の把握** — 既存の設計書・関連資料の状態を確認
3. **リスクの特定** — 潜在的な論点やブロッカーを洗い出し
4. **ステッププランの作成** — 設計ワークフローのフェーズに沿って分解
5. **確認を待つ** — 進める前にユーザーの承認を必ず受ける

## 使い方

```
/plan                              # 会話コンテキストから計画対象を推定
/plan 認証セッション管理を進めたい  # 具体的な要求を指定
```

## 動作方法

### 1. リクエストを分析

- `$ARGUMENTS` またはコンテキストから設計対象を特定
- 関連する Issue / 要求文書 (`docs/requirements/`) があれば確認
- 対象の設計書 (`docs/specs/`) が既に存在するか確認

### 2. 現状を評価

- 既存の設計書のどのフェーズまで完了しているか
- TODO / TBD / 未定 の数
- 論点テーブルの未解決数
- 関連する `docs/context/` の記載状況

### 3. 設計ワークフローに沿ったフェーズ分解

以下のフェーズから、残っている作業を特定 (詳細は `.rulesync/rules/design-workflow.md`):

| # | フェーズ | 完了条件 | 対応スキル |
|---|---|---|---|
| 1 | 起票 | 要件が把握できた | `/spec-create` |
| 2 | 下書き | 関連資料の解釈が記載された | `/spec-create` |
| 3 | 規約確認 | 規約が設計書に反映された | `/check-convention` |
| 4 | 論点解決 | 未確定事項ゼロ | `/spec-resolve` |
| 5 | DB 設計 | DDL が書ける状態 | (手動記載) |
| 6 | API 設計 | 全エンドポイントが定義された | (手動記載) |
| 7 | 認可設計 | 認可マトリクスが完成 | (手動記載 + マトリクス突合) |
| 8 | 図の生成 | 全ドメイン操作の図がある | `/spec-diagrams` |
| 9 | テスト設計 | テスト観点が揃った | `/spec-tests` |
| 10 | 差分整理 | 差分セクションが最新 | `/spec-track` |

### 4. リスクを評価

- 既存テーブル / API との破壊的変更の有無
- 認可要件の複雑さ (`docs/context/authorization/matrix.md` との整合)
- サンプルアプリ (go-react / kotlin-nextjs) への影響範囲
- 未確定の要件や不明瞭な仕様

### 5. 複雑度を推定

- **高**: 新規テーブル 3 個以上、API エンドポイント 5 個以上、認可パターンが複雑
- **中**: 新規テーブル 1-2 個、API エンドポイント 2-4 個
- **低**: 既存テーブルの拡張のみ、API エンドポイント 1 個

### 6. 計画を提示

フェーズ順に、具体的な作業内容と対応スキルを提示し、ユーザーの承認を待つ。

**CRITICAL**: 承認を受け取るまで設計書の変更は一切行わない。

## 他のスキルとの連携

- 計画承認後 → 各フェーズのスキルを順次実行 (`/spec-full` で自動化可)
- 途中で方針変更 → `/plan` を再実行して再計画
- レビュー指摘後 → 影響するフェーズを再計画
