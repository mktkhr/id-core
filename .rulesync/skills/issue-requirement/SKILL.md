# 要求整理 → Issue 起票スキル

ユーザーの要求を対話で整理し、Issue テンプレートに沿った本文を生成して起票する。

## ワークフロー

```
要求のヒアリング → 構造化 → ドラフト提示 → ユーザー承認 → Issue 起票
```

## 手順

### 1. 要求のヒアリング

`$ARGUMENTS` またはユーザーの発言から要求を把握する。
不足している情報があれば質問する:

- 誰がどう使うか
- なぜ必要か (背景・課題)
- 完了条件は何か
- 影響範囲 (`core/` か `examples/...` か)

### 2. テンプレート選択

要求の性質に応じて Issue テンプレートを選ぶ。
テンプレートが未整備の場合は本文を直接生成する。

| 要求の性質           | テンプレート (任意)                     | ラベル          |
| -------------------- | --------------------------------------- | --------------- |
| 新しい機能がほしい   | `.github/ISSUE_TEMPLATE/feature.md`     | `種別:機能追加` |
| 既存機能を改善したい | `.github/ISSUE_TEMPLATE/feature.md`     | `種別:改善`     |
| 企画段階で検討中     | `.github/ISSUE_TEMPLATE/requirement.md` | `種別:機能追加` |
| 調べたいことがある   | `.github/ISSUE_TEMPLATE/spike.md`       | `種別:調査`     |

### 3. ドラフト作成・提示 (必須 — スキップ禁止)

本文を作成し、**必ずユーザーに全文を提示する**。
ラベルの提案 (種別 + 優先度 + 対象) も併記する。

> **ガード**: ドラフトをユーザーに提示せずに起票してはならない。
> ユーザーが「そのまま起票して」と言った場合でも、ドラフトを表示した上で起票する。

### 4. マイルストーン特定 (必須)

詳細は `.rulesync/rules/issue-traceability.md` 参照。

1. 既存マイルストーン一覧を取得:

   ```bash
   gh api repos/{owner}/{repo}/milestones --jq '.[] | "\(.number) \(.title)"'
   ```

2. 要求のスコープからマイルストーンを推定し、ユーザーに確認:
   - 「この要求は M{N}.{X} に紐付けますか?」
   - 不明な場合は推測しない。ユーザーに選択を委ねる
   - 該当マイルストーンが存在しない場合は新規作成を提案する (スキルが勝手に作らない)

### 5. Issue 起票

ユーザーの承認後、`gh issue create` で起票する。

**必須項目**:

- 種別ラベル (上記対応表より)
- **`状態:要求分析中`** (起票=要求分析開始の宣言として自動付与)
- **マイルストーン** (必須、紐付けなし起票は禁止)
- 優先度・対象ラベル (任意)

```bash
gh issue create \
  --title "{タイトル}" \
  --body "{本文}" \
  --milestone "M{N}.{X}: {マイルストーンタイトル}" \
  --label "種別:機能追加" \
  --label "状態:要求分析中"
```

起票後、Issue 番号と URL をユーザーに報告する。

## 注意

- Milestone 未設定の場合は付けない (後から付ける運用)
- `状態:要求分析中` ラベルは起票=要求分析開始の宣言。
  `requirements-publish` 成功時に `状態:設計着手OK` に切り替わる
- リポジトリにラベルが未作成なら、初回のみ `gh label create` で作っておく
