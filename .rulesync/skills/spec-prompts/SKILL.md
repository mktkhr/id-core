# 実装プロンプト生成スキル

## 絶対ガード (違反時は生成を中止してやり直し)

**生成した全てのプロンプトに、以下のセクションが漏れなく含まれていなければならない。
1 つでも欠けていたら、そのプロンプトは未完成であり書き直す。**

| #   | 必須セクション              | 省略不可の理由                                                                             |
| --- | --------------------------- | ------------------------------------------------------------------------------------------ |
| 1   | `## 絶対ルール`             | テンプレート全文をそのまま転記。探索禁止・Codex レビュー必須を実行者に伝達する唯一の手段   |
| 2   | `## 作業ステップ`           | 各ステップに Codex レビュー手順 (番号付き) を含めること                                    |
| 3   | `## 実装コンテキスト`       | `${CONTEXT_DIR}` 変数定義 + 参照ファイル列挙。探索誘発表現 (「既存コードを参照」等) は禁止 |
| 4   | `## 前提条件`               | 依存先・後続タスクを明記                                                                   |
| 5   | `## 不明点ハンドリング`     | 矛盾・欠落時に停止してユーザーに質問する指示                                               |
| 6   | `## タスク境界`             | 実装する範囲 / しない範囲を明記                                                            |
| 7   | `## 設計仕様`               | 設計書から該当セクションを必要最小限で抜粋                                                 |
| 8   | `## テスト観点`             | 該当するテストケースのみ                                                                   |
| 9   | `## Codex レビューコマンド` | context 参照付きのレビューコマンド                                                         |
| 10  | `## 完了条件`               | チェックリスト形式。実行者はこれを最初にタスク化してから作業開始する                       |

## 完了条件のタスク化ルール

生成するプロンプトの `## 絶対ルール` セクション末尾に、以下の指示を必ず含める:

```
- **完了条件のタスク化**: 作業開始前に「完了条件」セクションの各項目を TaskCreate で登録し、
  各ステップ完了時に TaskUpdate で状態を更新すること。タスク化せずに作業を開始することは禁止
```

---

完成した設計書から、後続の実装セッション用プロンプトを Markdown ファイルとして生成する。

## ルールとテンプレート (必ず読む)

- `references/prompt-rules.md` — 分割・context 参照・TDD・Codex レビューのルール
- `references/prompt-template.md` — プロンプトの構成テンプレート

## 入力

- `$ARGUMENTS` から対象の設計書パスを特定する
- 引数なし: 会話コンテキストから対象設計書を推定する

## 出力先

```
docs/specs/{N}/prompts/
├── P1_01_*.md    # フェーズ 1: 並列不可の基盤作業
├── P2_01_*.md    # フェーズ 2: P1 完了後
└── ...
```

## 命名規則

```
P{フェーズ番号}_{連番}_{対象}_{スコープ}.md
```

例:

- `P1_01_core_db.md` — core/ の DB 基盤
- `P2_01_core_oidc_token_endpoint.md`
- `P3_01_go_react_frontend_login.md`

## 生成手順

1. `prompt-rules.md` と `prompt-template.md` を読み取る
2. AskUserQuestion で実装ツールを確認 (デフォルト: Codex CLI)
3. 設計書を全文読み取り
4. ドメイン操作 (Op) の依存関係を分析
5. ツールに応じた粒度で分割 (rules 参照)
6. 各プロンプトを template に従って生成
7. **セルフ検証**: 生成した各プロンプトを絶対ガードの 10 項目と突合し、欠落があれば書き直す
8. 生成したファイル一覧と依存関係図を報告

## 生成後の報告

```markdown
| ファイル    | フェーズ | 並列            | 依存先 | 概要 |
| ----------- | -------- | --------------- | ------ | ---- |
| P1*01*\*.md | 1        | -               | なし   | ...  |
| P2*01*\*.md | 2        | P2_02 と並列 OK | P1_01  | ...  |
```

## 自己完結ルール

実装プロンプトは**自己完結型**とする (`.rulesync/rules/path-resolution.md` 準拠):

- 設計書からの引用を基本とする
- 他ファイル (context 等) への参照は最小限
- 他リポジトリへの参照は禁止
- このリポジトリ外の絶対パスを書かない
