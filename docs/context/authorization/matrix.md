# 認可マトリクス

> 最終更新: 2026-04-25 (骨格セットアップ時点、内容 TBD)

このドキュメントは **id-core の認可の唯一の正本**。

設計書 (`specs/*/index.md`) の認可表は本ファイルの**参照コピー**であり、独自判断で拡大/縮小してはならない。マスターと差分を検知した場合、設計・実装系スキルは**即停止してユーザー判断を仰ぐ** (推測禁止)。

## ロールセット (TBD)

最初の要求 (`requirements/1/`) を起こすタイミングで、id-core の対象ユーザー・対象プロダクトに即して定義する。

候補となる軸 (現時点での仮説、確定ではない):

| 軸 | 例 |
|---|---|
| 利用者の所属・属性 | 例: 組織所属ユーザー / 一般ユーザー |
| 認証手段 | 上流 IdP 認証済 / 電話番号認証済 / SNS 認証済 |
| プロダクト管理権限 | プロダクト管理者 / 一般利用者 |
| id-core 自体の管理権限 | id-core 管理者 / id-core 運用者 |

## 機能 × ロールマトリクス (TBD)

各機能は最初の要求・設計フェーズで洗い出し、ここに正本として登録する。

凡例:
- `o` = 利用可能
- `▲` = 自分のデータのみ (条件付き許可)
- 空欄 = 利用不可

## 運用ルール

### 推測による変更禁止

- マスターの空欄セル (拒否) を `o` / `▲` に拡大する提案は**ユーザー判断必須**
- マスターの `o` セルを縮小する提案も**ユーザー判断必須**
- マスター未記載の機能・ロールを追加する場合も**ユーザー判断必須**

### マスター変更の正規フロー

マスター変更が必要と判断された場合 (ユーザー承認後のみ):

1. 設計書に「**認可マスター変更提案**」セクションを設け、差分 (before/after)・理由・影響範囲を記載
2. ADR を起票 (`docs/adr/`)
3. ユーザー承認後にマスター (本ファイル) を更新
4. 更新後のマスターをもとに認可ポリシー実装を再生成
