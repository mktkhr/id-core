# 要求 #21: DB 接続 + マイグレーション基盤を core/ に導入

- 関連 Issue: [#21](https://github.com/mktkhr/id-core/issues/21)
- マイルストーン: [M0.3: DB 接続 + マイグレーション基盤](https://github.com/mktkhr/id-core/milestone/3)
- 状態: 設計着手OK
- 起票日: 2026-05-02
- 最終更新: 2026-05-02

## 背景・目的

M0.2 までに `core/` には HTTP サーバー骨格 + 構造化ログ + エラーハンドリング基盤が乗ったが、永続化層が存在しない状態にある。後続マイルストーンで以下の DB 依存機能が控えている:

- M2.1 (クライアント登録 DB)
- M2.2 (認可コード / refresh token の DB 化)
- M3.1 (ユーザー DB + 共通 ID 発行)
- M7.1 (監査ログ永続化)

特に M2.x / M3.x / M7.1 では「ユーザー / クライアント / トークン / 監査イベント」と性質の異なるテーブルが順次積み上がるため、マイグレーション追加頻度が高い見込みである。これらを各マイルストーンで個別に DB 接続 / マイグレーション / 接続規約を作り直すと、**毎回違う流儀で DDL が積まれて運用負荷が雪だるま式に増える**。本マイルストーンでは「`core/` から DB に接続して migrate up/down が動き、後続マイルストーンが乗れる規約が固まっている」状態を作る。

本要求のスコープは**基盤の確立と規約整備**であり、実テーブル (アカウント / クライアント / トークン等) は対象外。最低限のスモークテーブル (例: `schema_migrations` 自動管理 + 動作確認用 1 テーブル) があれば十分とする。

## ユーザーシナリオ

開発者・運用者として、

1. `docker compose up` でローカルに PostgreSQL コンテナが立ち上がる
2. リポジトリの make コマンド一発でマイグレーション up が走り、down も同様に走る
3. `core/` のサーバを起動すると DB 接続が確認され、接続失敗時は明示エラーで起動失敗する
4. CI 上でも同じ手順でマイグレーションが回り、ローカルとの再現性が保たれる
5. 後続マイルストーンの実装担当が「マイグレーションファイル命名規則 / トランザクション境界 / 接続文字列の組み立て方」を本要求由来の規約書から確認できる

これにより、本マイルストーン以降の DB 依存機能は **既存の接続スタック + マイグレーション運用に乗るだけで実装に着手できる**状態になる。

## 用語

- **要求文書**: 本ファイル (`docs/requirements/21/index.md`)
- **規約書**: 設計フェーズで生成する `core/` 向けの DB / マイグレーション運用規約。`docs/context/backend/conventions.md` の DB / マイグレーション節 (現状 TBD) を埋める想定 (格納場所は `/spec-resolve` で決定)
- **スモークテーブル**: 本マイルストーンで作る最低限の動作確認用テーブル。実ドメイン情報を持たず、マイグレーション up/down が動くことを示すためだけに存在する

## 機能要件

| ID   | 内容                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ---- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| F-1  | `core/` から DB (PostgreSQL を第一候補とし、最終決定は `/spec-resolve`) に接続できる。接続情報 (`host` / `port` / `user` / `password` / `dbname` / `sslmode`) は環境変数で設定する。命名スキームは `CORE_DB_<NAME>` 形式の個別変数方式を第一候補とし、単一 DSN 変数方式 (`CORE_DATABASE_URL` 等) との優先順位は Q7 で確定。ただしいずれの方式を採っても **パスワードは他の接続要素から独立した env で渡せること**を最低ラインとして固定する (シークレット分離管理のため)                                                                                                    |
| F-2  | マイグレーションツール (`golang-migrate` を第一候補とし、`atlas` / `goose` 等の代替評価は `/spec-resolve`) で up / down が動く。マイグレーションファイルの配置先は `core/db/migrations/` を第一候補とし、最終位置は `/spec-resolve` で確認 (M0.1〜M0.2 では DB を扱わなかったため新設)                                                                                                                                                                                                                                                                                      |
| F-3  | マイグレーション up を実行する Make ターゲット (例: `make migrate-up`) と down を実行するターゲット (例: `make migrate-down`) が `core/Makefile` に追加される。ターゲット名・引数の最終形は `/spec-resolve`                                                                                                                                                                                                                                                                                                                                                                 |
| F-4  | マイグレーションファイルの命名規則 (例: `{連番 8 桁}_{スネークケース slug}.{up\|down}.sql`) が固定され、規約書に記載される。連番衝突時の解決方針も明記する                                                                                                                                                                                                                                                                                                                                                                                                                  |
| F-5  | 各マイグレーションファイルの **トランザクション境界の規約** が固定される。デフォルトは「1 ファイル = 1 トランザクション」を第一候補とし、CREATE INDEX CONCURRENTLY 等のトランザクション内不可文を使う場合の例外規約を明記する                                                                                                                                                                                                                                                                                                                                               |
| F-6  | 起動時に DB 接続が確認され、接続失敗時は `logger.Error` でエラーを記録した上で**非ゼロ終了コード**でプロセスを終える (`log.Fatal*` 不使用ポリシーは踏襲)。具体的な終了処理 (`os.Exit(1)` 直呼び出し / graceful shutdown フック経由 / cmd 層の `runErr` 集約) は `/spec-resolve` で確定                                                                                                                                                                                                                                                                                      |
| F-7  | `/health` 系エンドポイントで **DB 接続状態が確認できる**。HTTP レスポンス形式 (status code / JSON 構造 / エンドポイント分割) の最終形は Q6 で `/spec-resolve` 決定。OIDC OP の **可用性監視の最低ライン**として、DB 切断時は監視ツール (k8s probe 等) が異常を検知できる挙動 (status code or JSON フラグ) のいずれかを採る方針のみを固定する。**公開する情報の粒度下限は「DB 接続可否のみ」**とし、DB 製品名・バージョン・ホスト名・接続文字列・内部エラーメッセージ・スタックトレース等の運用詳細は `/health` 経路で外部に露出しない (詳細情報は内部 ERROR ログにのみ記録) |
| F-8  | DB クライアントライブラリ (`database/sql` + `lib/pq` / `database/sql` + `pgx/v5` driver / `pgx/v5` direct / `sqlx`) の選定が `/spec-resolve` で確定する。第一候補は標準 `database/sql` + `pgx/v5` driver 系。本要求段階では「ライブラリ選定 = 設計フェーズ送付」のみを固定する。**sqlc によるクエリコード自動生成の本格運用はスコープ外** (M2.x 以降)、ただしライブラリ選定時は将来の sqlc 連携可否を採用基準の 1 つとして考慮する                                                                                                                                          |
| F-9  | コネクションプール設定 (`MaxOpenConns` / `MaxIdleConns` / `ConnMaxLifetime` / `ConnMaxIdleTime`) が環境変数または config で設定可能。既定値は規約書に明記する                                                                                                                                                                                                                                                                                                                                                                                                               |
| F-10 | DB 接続情報 (パスワード等のシークレット) は M0.2 の redact deny-list (`docs/context/backend/conventions.md` 「redact 対象キー一覧」) と整合する形でログに出さない。具体的にはエラーログ上での DSN / 接続文字列のフルダンプを禁止し、host / db_name 等の **シークレットを含まない要素のみログに出してよい**                                                                                                                                                                                                                                                                  |
| F-11 | 開発環境用の PostgreSQL コンテナが `docker/compose.yaml` で起動できる。コンテナは `core/` のテストおよびローカル動作確認に利用する。テスト用 DB の初期化方針 (毎回 drop & recreate / 永続化) は `/spec-resolve` で確定                                                                                                                                                                                                                                                                                                                                                      |
| F-12 | マイグレーション実行と CI の整合: **DB を要するテスト実行前に migrate up が完了している**状態で `make test` が走り、ローカルと CI で同一の migrate 済み DB に対してテストが実行できる。具体的な CI 構成 (job 分割 / service container / migrate 起動順序) は `/spec-resolve` で決定                                                                                                                                                                                                                                                                                         |
| F-13 | マイグレーション失敗時の挙動と運用導線が固定される。デフォルトは「migrate up 失敗 → 次の起動を阻止 (start gate)」を第一候補とする。自動 down rollback は採用しない方針 (人手判断を要するため)。失敗時の手動復旧手順 (旧バージョン image rollback / 手動 down → 修正 → 再 up / `schema_migrations` の dirty 状態解消) は規約書に明記する (具体手順は `/spec-resolve`)                                                                                                                                                                                                        |
| F-14 | スモークテーブル (本マイルストーン専用、実ドメイン情報を持たない) が 1 つ以上存在し、double-roundtrip (up → down → up → down) で migrate 0 状態に戻ることを確認する整合テスト (Go ユニットテスト + CI 上の migrate 実行) が用意される。合格判定の最低条件: (a) 各マイグレーションファイルで作成した object (table / index / constraint) が down 後に消える、(b) `schema_migrations` テーブルの記録が roundtrip 後に initial 状態と一致する、(c) double-roundtrip 全工程でエラー終了しない                                                                                   |
| F-17 | テスト間の状態隔離が確保されている。具体的な隔離方式 (毎テスト drop & recreate / tx-rollback / テストごとに独立 schema 等) は Q8 で `/spec-resolve` 決定だが、**最低ライン**として「並列実行 (`go test ./...` のデフォルト並列度) で互いの state を汚染しない」「テスト失敗後に残留 state が次のテスト実行をブロックしない」の 2 点を要件として固定する                                                                                                                                                                                                                     |
| F-15 | `core/` で `make build && make test && make lint` が pass する (M0.1〜M0.2 から踏襲)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| F-16 | 規約書 (本要求で生成する `core/` 向け DB / マイグレーション運用規約) が存在する。最低必須項目: (1) DB 製品・バージョン・ライブラリ・マイグレーションツール選定結果、(2) 環境変数一覧 (`CORE_DB_*`)、(3) マイグレーション命名規則と例、(4) トランザクション境界の規約、(5) 開発者向け運用手順 (compose / make / CI)                                                                                                                                                                                                                                                          |
| F-18 | DB 接続層・マイグレーション実行・クエリ実行の各経路は `context.Context` を引数で受け取り、`logger.RequestIDFrom(ctx)` / `logger.EventIDFrom(ctx)` で取り出した相関 ID をログレコードに付与できる構造とする。M0.2 の F-14 (Domain 層からロガーを直接呼ばない / context から取り出すのみ) と D1 順序の context 伝播規約を踏襲し、DB 関連ログも request_id / event_id によって追跡可能にする                                                                                                                                                                                   |

## 非機能要件

| 区分           | 要件                                                                                                                                                                                                                                                                                                                                                                               |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| パフォーマンス | M0.3 では計測対象外。コネクションプール既定値は「Phase 0 の検証用 PoC として過小ではないが過大でもない」レベルで仮置きし、SLO は M1.x 以降で定義                                                                                                                                                                                                                                   |
| セキュリティ   | DB パスワード等は環境変数経由のみ受け取り、コードベースに直書きしない。開発用 compose の既定パスワードはダミー扱いで本番禁止であることを規約書に明記する。**SSL/TLS の最低ライン**: 本番想定環境では平文接続 (`sslmode=disable`) を許容しない (具体的な mode = `require` / `verify-ca` / `verify-full` の選定は Q10 で `/spec-resolve`)。開発・テスト環境では `disable` を許可する |
| 可用性         | 起動時 DB 切断は **明示エラーで起動失敗** する (隠蔽しない)。実行中の一時切断時の挙動 (リトライ / circuit breaker) は M1.x 以降で扱う想定                                                                                                                                                                                                                                          |
| 保守性         | マイグレーション・接続・プール設定は単一の責務に分離する (`internal/db` / `internal/dbmigrate` 等のパッケージ境界は `/spec-resolve` で決定)                                                                                                                                                                                                                                        |
| ポータビリティ | OS / アーキテクチャ依存コードを書かない。マイグレーション CLI もバイナリ依存より Go パッケージとしての利用を優先する (例: `golang-migrate/migrate/v4` のライブラリ API)                                                                                                                                                                                                            |
| 再現性         | ローカル / CI で同一の migrate up/down 結果が得られる。docker compose の DB バージョンは pin (例: `postgres:17.x`) し、`latest` 等の浮動 tag は使わない                                                                                                                                                                                                                            |
| 検証性         | F-1〜F-14 を Go 単体テストおよび最低 1 本の統合テスト (compose 由来 DB に対する up/down round-trip) で検証可能。CI 上で同等のテストが green になる                                                                                                                                                                                                                                 |

## スコープ外 (含まないこと)

- 実テーブル DDL (アカウント / クライアント / トークン / ユーザー等) は本マイルストーンで作らない (→ M2.1 / M2.2 / M3.1)
- 監査ログ永続化 (→ M7.1)
- レプリケーション / バックアップ / リカバリ運用 (本リポジトリは PoC、本番運用は別リポジトリ)
- DB at-rest 暗号化 / TDE 設定
- マイグレーションの高度なツール (zero-downtime migration / blue-green schema 等)
- sqlc / Repository コード自動生成の本格運用 (本要求ではライブラリ選定の文脈で言及するに留める。実利用は M2.x 以降で本格化)
- DB レベルの監視 (`pg_stat_statements` / Prometheus exporter 等)
- 複数 DB スキーマ / マルチテナント設計
- マイグレーション履歴の改竄検知 (`schema_migrations` テーブルのチェックサム強制)

## 認可要件

本要求は**認可スコープ外**である。

理由:

- 本要求は DB 接続層 / マイグレーション基盤を導入するもので、エンドポイントの認可可否を変更しない
- `/health` の DB チェック追加 (F-7) は既存の **認証なし公開エンドポイント** に DB 状態を反映するだけで、ロール別の閲覧制限を新設しない
- 認可マスター [`docs/context/authorization/matrix.md`](../../context/authorization/matrix.md) のセル変更を伴わない

> 認可マトリクス突合: 該当セルなしを確認済み (突合不要)。判定根拠は上記 3 点と、本要求が新規認可対象エンドポイントを追加しないこと (`/health` 系のみが対象) に基づく。
