# 要求 #7: ログ・エラー・request_id の規約を確定し core/ に組み込む

- 関連 Issue: [#7](https://github.com/mktkhr/id-core/issues/7)
- マイルストーン: [M0.2: ログ・エラー・request_id の規約確定](https://github.com/mktkhr/id-core/milestone/2)
- 状態: 設計着手OK
- 起票日: 2026-05-02
- 最終更新: 2026-05-02

## 背景・目的

M0.1 で `/health` の最小サーバー骨格が立ち上がったが、ロギングは標準 `log` パッケージ + `log.Fatalf` のみで構造化されておらず、エラーレスポンス規約も未定義の状態にある。

ログ・エラー規約は M1.x (OIDC エンドポイント) / M0.3 (DB) / M4.x (上流連携) など以降の全コンポーネントが乗る土台であり、最初期に確定しないと後で剥がす負債が大量に発生する。
障害時に「どのリクエストで何が起きたか」を追跡できないと運用が成立しないため、`request_id` を全構造化ログ・全エラー応答に串刺しできる仕組みを Phase 0 のうちに導入する。

また、本要求では `backend-logging` 規約の「Domain 層でログを出さない」原則と、`backend-security` 規約の「シークレットをログに出さない (deny-list redact)」原則を最初の段階から守らせる。

## ユーザーシナリオ

開発者・運用者として、id-core にリクエストを投げると、

1. リクエストごとに一意な `request_id` が割り振られ、レスポンスヘッダ `X-Request-Id` と関連する全構造化ログレコードに同じ値が含まれる
2. クライアントが `X-Request-Id` ヘッダを付けて送ってきた場合、その値が**妥当性検証を通過したときに限り**そのまま採用される。妥当でない値は破棄して再生成し、元値はサニタイズした上で別フィールドに記録する
3. 4xx / 5xx 応答時には規約通りの JSON 形式でエラーが返り、同じ `request_id` がログにも記録される。スタックトレース等の内部情報はクライアントへ返さない
4. ログ出力先は標準出力で、本番想定では JSON Lines、開発時には人間可読フォーマットへ切り替えできる

これにより「クライアントから渡された `request_id` でログを grep すれば原因が特定できる」状態と、「画面/CLI で見たエラーレスポンスから即座にログ追跡できる」状態が両立する。

## 機能要件

| ID   | 内容                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ---- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| F-1  | 構造化ログが標準出力に JSON Lines 形式で出力される (本番想定モード)。クライアント由来文字列はすべて JSON エンコーダ経由で出力し、改行・制御文字によるログインジェクションを防ぐ                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| F-2  | 開発時用に人間可読フォーマットへ切り替えるモードが用意されている (環境変数で切替、具体名は `/spec-resolve` で決定)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| F-3  | HTTP 経路で発生する全ログレコードに `request_id` を必須付与する。最低限のフィールドは `time` (UTC・ISO8601 系、具体形式は `/spec-resolve`)、`level` (DEBUG/INFO/WARN/ERROR)、`msg`、`request_id`、`method` / `path` / `status` / `duration_ms` (アクセスログ系)。追加フィールドは設計で決定                                                                                                                                                                                                                                                                                                                                                                                                                       |
| F-4  | HTTP 経路外のログレコード (起動 / signal handler / バックグラウンドジョブ) には HTTP 由来の `request_id` を入れず、代わりに `event_id` (起動毎・ジョブ毎に発番される一意 ID、形式は `/spec-resolve`) を必須付与する。これにより HTTP 内外いずれのログも一意キーで追跡可能となる                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| F-5  | HTTP middleware で `request_id` を生成 (UUID v4 を第一候補とし、最終決定は `/spec-resolve`) し、`context.Context` に伝播する。レスポンスヘッダ `X-Request-Id` を必ず返す (4xx / 5xx / panic 時を含む)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| F-6  | クライアント提供 `X-Request-Id` の妥当性基準を以下に固定する: (a) 長さ最大 128 オクテット以内、(b) 文字種は ASCII 印字可能文字のみ (バイト範囲 `0x21`–`0x7E`、制御文字・改行 (`0x0A` / `0x0D`)・空白 (`0x20`)・タブ (`0x09`) を含まない)、(c) 上記を満たす場合のみ採用、満たさない場合は破棄して新規生成する。破棄した元値はサニタイズ (制御文字を `\u00XX` に置換、長さ切詰め) の上、ログにのみ別フィールド `client_request_id` として残す                                                                                                                                                                                                                                                                       |
| F-7  | 内部 API (id-core 自身の管理 API・healthcheck 系) のエラーレスポンス JSON 形式は `{ "code": string, "message": string, "details"?: object, "request_id": string }` を基本形とする。`details` は object / array に限定し、シークレット (F-13) を含めてはならない。最終形式 (RFC 7807 採用可否含む) は `/spec-resolve` で決定するが、本基本形を逸脱する場合は ADR を起票する                                                                                                                                                                                                                                                                                                                                        |
| F-8  | OIDC / OAuth 2.0 エンドポイント (M1.x 以降の `/authorize` / `/token` / `/userinfo` 等) のエラーレスポンスは RFC 6749 / OIDC Core / RFC 6750 準拠 (`error` / `error_description` / `error_uri`) とすることを方針として確定する。実装は M1.x で行う。F-7 (内部 API 形式) と F-8 (OIDC 形式) の使い分け境界はエンドポイント単位で固定し、混在を禁止する                                                                                                                                                                                                                                                                                                                                                              |
| F-9  | エラーハンドラ middleware が、(a) `panic`、(b) 既知のドメインエラー、(c) 未捕捉の `error` の 3 系統を全て規約 JSON で返す。エラーレスポンス JSON には必ず `request_id` 相当のフィールドを含む (F-8 の RFC 6749 系では `error_uri` 以外の補助フィールドで運ぶ方針を `/spec-resolve` で決定)                                                                                                                                                                                                                                                                                                                                                                                                                        |
| F-10 | panic 発生時はクライアントへ固定メッセージ + `request_id` のみを返す。スタックトレース・内部ファイルパス・実装詳細を HTTP レスポンスへ絶対に含めない。スタックトレースは内部の構造化ログ (level=ERROR) にのみ記録する                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| F-11 | 各種失敗ケース (panic / 4xx / 5xx) で、規約に沿った構造化ログが残る。最低ルール: 5xx および panic は `level=ERROR`、4xx は `level=WARN` 以下とし、原因の error chain を構造化フィールドに含める。詳細なレベル使い分けガイドは `/spec-resolve` で確定                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| F-12 | M0.1 由来の `log.Fatal*` (`Fatalf` / `Fatalln` / `Fatal`) を `core/` 配下から完全に排除する。完了条件は `grep -rn "log\.Fatal" core/` の出力が 0 件であることで判定する。新規追加コードでも禁止                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| F-13 | シークレットはログ出力時に redact される。対象キーの deny-list は最低限以下を含む: `Authorization` / `Cookie` / `Set-Cookie` ヘッダ、`password` / `access_token` / `refresh_token` / `id_token` / `code` / `client_secret` / `assertion` / `client_assertion` フィールド。適用面は HTTP リクエストヘッダ・JSON body・query / form パラメータ・エラー `details`・error chain メッセージの全てとする。キー照合規則: HTTP ヘッダ名は **case-insensitive**、JSON body / form / query パラメータ・`details` のキーは **case-insensitive かつ完全一致** (部分一致は誤検知を生むため禁止)、ネストした JSON / map は **再帰的に走査** して全階層に適用、配列要素も同様に走査。完全な対象キー一覧は `/spec-resolve` で確定 |
| F-14 | `Domain` 層 (=ビジネスロジック層) でロガーを直接呼び出さない。ログ出力は middleware / handler / infrastructure 層が担当し、その際に `context.Context` から `request_id` (または `event_id`) を取得して付与する。Domain 層が返す `error` には `request_id` を埋め込まず、上位層で context から付ける (`backend-logging` 規約)                                                                                                                                                                                                                                                                                                                                                                                      |
| F-15 | `core/` で `make build && make test && make lint` が pass する                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| F-16 | ログスキーマ契約テストが用意されている。検証対象は最低 2 系統に分割: (a) HTTP 経路系 — `time` / `level` / `msg` / `request_id` / `method` / `path` / `status` / `duration_ms` の存在と型 (最低 1 ケース)、(b) 非 HTTP 経路系 (起動 / signal handler / バックグラウンドジョブ) — `time` / `level` / `msg` / `event_id` の存在と型 (最低 1 ケース)。Go 単体テストとして CI で実行し、将来の破壊的変更を検知する                                                                                                                                                                                                                                                                                                     |
| F-17 | 上記規約をまとめたドキュメント (規約書) が存在する。最低必須項目は以下: (1) ログフィールド定義 (HTTP 系 / 非 HTTP 系の必須・任意フィールド表)、(2) エラーレスポンス境界の判定基準 (内部 API = F-7 / OIDC = F-8 のエンドポイント別マッピング表)、(3) redact 対象キーの完全一覧と適用面 (F-13 を反映)、(4) ログレベル使い分けガイド (F-11 を反映)、(5) 開発者向け運用手順 (環境変数で format 切替する手順、テスト時の確認方法)。検証方法はドキュメントレビュー (`/doc-review`) で必須項目の存在を確認、加えて F-16 の契約テストでフィールド定義との整合を確認。格納場所は `/spec-resolve` で決定                                                                                                                    |

## スコープ外 (含まないこと)

- DB アクセス時のクエリログ (sqlc 導入時に `/spec-resolve` で決定 → M0.3)
- 監査ログ・認可操作の永続化 (→ M7.1)
- メトリクス (Prometheus 等) / 分散トレーシング (OpenTelemetry) — `request_id` のフィールドは出すが、metric/trace exporter の連携は別マイルストーン
- ログのファイル出力 / ログローテーション (本リポジトリは検証用 PoC のため標準出力のみ)
- ログ集約基盤 (Loki / Datadog / CloudWatch 等) との接続
- OIDC エンドポイントの実エラーハンドリング実装 (方針宣言 F-8 のみ。実装は M1.x)
- `panic` の自動再起動 / 監視 (graceful shutdown は M0.2 で扱わない)
- 多言語化 (`error_description` の i18n) は OIDC 画面導入時の M3.3 から
- ログ drop カウンタの外部公開 (メトリクス導入時に再考)

## 認可要件

本要求は**認可スコープ外**である。

理由:

- 本要求はログ・エラー・`request_id` の横断規約と middleware を導入するもので、エンドポイントの認可可否を変更しない
- `/health` (M0.1) は認証なしで誰でもアクセスできる公開エンドポイントのまま
- ユーザー / クライアント / ロールという概念自体が M2 以降で導入される
- 認可マスター [`docs/context/authorization/matrix.md`](../../context/authorization/matrix.md) のセル変更を伴わない

> 認可マトリクス突合: 該当セルなしを確認済み (突合不要)。判定根拠は上記理由 4 点と、本要求が新規エンドポイントを追加しないこと (既存 `/health` のみが対象) に基づく。
