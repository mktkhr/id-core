# 要求 #1: core/ の最小 HTTP サーバーを起動できるようにする

- 関連 Issue: [#1](https://github.com/mktkhr/id-core/issues/1)
- マイルストーン: [M0.1: core/ の最小 HTTP サーバー](https://github.com/mktkhr/id-core/milestone/1)
- 状態: 設計着手OK
- 起票日: 2026-05-02
- 最終更新: 2026-05-02

## 背景・目的

id-core はモノレポ内で `core/` ディレクトリだけが空の状態。Phase 1 以降の OIDC OP 実装は全てこの骨格上に乗るため、最初に「ビルドできて起動できてヘルスチェックが通る」状態を確立する必要がある。

本要求は Phase 0 (スパイク) の起点であり、後続マイルストーン (M0.2 ログ規約 / M0.3 DB 接続 / M1.x OIDC 実装) の前提となる土台を作る。

## ユーザーシナリオ

開発者として、`core/` ディレクトリで `make build` → 生成されたバイナリ起動という単純な手順で HTTP サーバーが立ち上がり、`/health` への GET リクエストに `200 OK` を返してほしい。これにより、後続マイルストーンを安全に積み上げられる。

## 機能要件

| ID  | 内容                                                                                                                                                                                                                       |
| --- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| F-1 | `core/` 配下に Go モジュール (`go.mod`) が初期化されている                                                                                                                                                                 |
| F-2 | `core/Makefile` に `build` ターゲットがある                                                                                                                                                                                |
| F-3 | `make -C core build` が成功し、設計で決定した出力パスにバイナリが生成される (具体パスは `/spec-resolve` で決定)                                                                                                            |
| F-4 | バイナリ起動後、`GET /health` に対して以下を返す: HTTP `200`、`Content-Type: application/json`、必須フィールド `{"status":"ok"}` を含む JSON ボディ (追加フィールドは設計で決定)                                           |
| F-5 | サーバーのリッスンポートが環境変数 `CORE_PORT` で変更できる。具体的なデフォルト値は `/spec-resolve` で決定。**未設定時はデフォルト値で起動、不正値 (非数値・範囲外 0〜65535 を逸脱) の場合は明示エラーを出力して起動失敗** |
| F-6 | ビルド・起動・ヘルスチェック確認の手順が `core/README.md` に記載されている。前提 (Go バージョン)・実行コマンド・期待される最小出力例 (`HTTP 200 / {"status":"ok"}`) を含む                                                 |
| F-7 | `/health` の応答 (HTTP 200 + 必須 JSON フィールド) を自動検証する仕組み (Go の単体テストまたは `make -C core test` 等) が用意されている                                                                                    |

## 非機能要件

| 区分           | 要件                                                                                                              |
| -------------- | ----------------------------------------------------------------------------------------------------------------- |
| パフォーマンス | M0.1 では計測対象外 (Phase 1 以降で SLO を定義)                                                                   |
| セキュリティ   | `/health` は認証不要・公開エンドポイント (情報漏えい防止のためレスポンスにバージョン情報以外の機密情報を含めない) |
| 可用性         | M0.1 では単一プロセス起動で可。冗長化・graceful shutdown は M0.2 以降で別途検討                                   |
| 保守性         | 標準ライブラリ + `net/http` (または同等) を基本とし、ルーティングライブラリ採用判断は M0.2 以降に延期             |
| ポータビリティ | OS 依存コードを書かない (Linux / macOS で動作することを前提)                                                      |
| 国際化         | レスポンスは英数字のみ (i18n 対応は OIDC 画面導入時の M3.3 から)                                                  |
| 再現性         | `core/README.md` の手順に従えばローカル / CI で同一結果が得られる (Go バージョンを README で固定明記)             |
| 検証性         | `/health` の応答が自動テストで検証可能 (F-7 の前提)                                                               |

## スコープ外 (含まないこと)

- 構造化ログ規約 (→ M0.2)
- エラーレスポンス共通規約・request_id 付与 (→ M0.2)
- DB 接続 / マイグレーション (→ M0.3)
- 認証・認可・OIDC エンドポイント (→ M1.x 以降)
- Docker 化 (M0.1 ではバイナリ直起動のみ)
- graceful shutdown / 監視 (M0.2 以降)

## 認可要件

本要求は**認可スコープ外**である。

理由:

- `/health` は認証なしで誰でもアクセスできる公開エンドポイントとする
- ユーザー / クライアント / ロールという概念自体が M2 以降で導入される
- `docs/context/authorization/matrix.md` (正本) のセル変更を伴わない

> 認可マトリクス突合: 不要 (該当セルなし)
